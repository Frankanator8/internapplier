# Scoring

Score the resume on a 0‚Äì10 scale (decimals allowed):
- 10 = exceptional match: every bullet quantified, tightly tailored to the JD, strong verbs, no fluff, clean layout.
- 9 = excellent: minor polish needed.
- 8 = good but with clear, fixable weaknesses.
- 7 or below = significant problems (vague bullets, missing JD keywords, weak structure, irrelevant content).

Reserve scores above 9.5 for resumes you would forward to a hiring manager with confidence on the first read.

Evaluate along these axes:
1. Relevance to the JD ‚Äî are the most relevant experiences and skills surfaced first? Are JD keywords woven in naturally?
2. Bullet quality ‚Äî strong action verbs, quantified impact, specific technologies, no filler.
3. Conciseness and density ‚Äî every line earning its space; no redundancy.
4. Skills section ‚Äî accurate, JD-aligned, not a keyword dump.
5. Structure and readability ‚Äî clear hierarchy, consistent formatting.
6. Recency (using `<today>`) ‚Äî recent work surfaced over stale work, when both are present and comparable.

# Feedback

The `feedback` field is consumed by the next generate pass, which owns BOTH bullet wording AND layout. Put everything actionable in one place: specific bullets to rewrite and how (reference them by content), missing JD keywords, what to quantify, weak action verbs to replace, section ordering, skills-section composition, layout density, structural issues. The next pass may reword existing bullets, reorder sections, adjust the skills list, and tighten layout ‚Äî but it may NOT invent facts or add new bullets.

Use "" if there is nothing actionable.

# Drops (page-fit protocol)

`drops` MUST be `[]` UNLESS `<page_status>` shows `fill > page_cap`. Do not propose drops on a fit-clean draft.

When over cap, populate `drops` with the items the writer should REMOVE on the next attempt, in priority order (weakest first). Each item is a string built from the resume JSON in one of these forms:

- An **entry label** built from the section item:
  - Experience: `{role} @ {company}` (e.g. `Software Engineer @ Acme`) ‚Äî built from `experience` section items' `role` and `company`.
  - Project: `{name}` (e.g. `Distributed Cache Library`) ‚Äî built from `projects` section items' `name`.
  - Award: `{title} ‚Äî {issuer}` using an em-dash (e.g. `Dean's List ‚Äî University of X`) ‚Äî built from `awards` section items.
- A **course name** drawn verbatim from any `education[].courses` entry. Courses are individually droppable.

Choose enough items to plausibly free `fill - page_cap` pages of space. For small overflows (e.g. `0.10`‚Äì`0.30`) a few course removals or a single weak bullet-heavy entry may suffice; for larger overflows prefer dropping weak/older entries. Use course removals for finer-grained trimming. Use `<today>` to identify stale items.

Match strings exactly as they appear in the resume ‚Äî the orchestrator does case-insensitive, whitespace-collapsed matching, but exact wording is safest. Unmatched strings are silently ignored.
